# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Scope

`meta-ai-agent` is a transport/orchestration package — Meta webhook ingestion, signature verification, outbound delivery, status tracking, dedupe, conversation buffering, and ordered delivery for an AI agent that runs over WhatsApp Cloud API, Facebook Messenger, and Instagram DMs under a single Meta App. The developer brings their own HTTP chat endpoint. The transport package itself stays model-provider-free: do not couple `src/` to a model provider, add LLM-specific assumptions to the chat contract, or pull OpenAI/Anthropic/etc. SDKs into the root `package.json`.

`meta-ai-agent-implementation-plan.md` is the durable project guidance (intent, ten-stage roadmap, Meta platform constraints, testing strategy, implementation rules). Read it before non-trivial changes — it lists API quirks (single-app/three-products webhook fan-in, channel-specific timestamp/typing/echo rules, messaging windows, retry behavior, token lifecycle) that the code intentionally relies on.

## Stage X status

**Stages 1–3 complete: scaffolding, signature verification, webhook routing, payload parsing, normalized message types, setup verification, payload capture. Stages 4–10 planned — see `meta-ai-agent-implementation-plan.md`.** No outbound clients (beyond the single per-channel verify smoke-tests), no conversation agent, no identity resolution, no metrics, no persistence yet. The POST `/webhook` handler today verifies the Meta signature, ACKs 200, parses the payload via `parseMetaWebhook`, and emits per-message / per-status / per-channel structured logs. The parsed `ParseResult` is discarded on the route path; Stage 5 will hand it to the conversation agent.

## Architecture

The runtime is a single Express app composed in `src/http/app.ts`. `createApp({ config, logger })` wires:

- `express.json({ limit: '5mb', verify })` first. The `verify` hook copies the raw body buffer onto `req.rawBody` **before** JSON parsing mutates it — signature verification depends on byte-exact body capture. This is load-bearing; do not introduce another body parser ahead of it without preserving `rawBody`.
- `GET /webhook` — Meta verification handshake. Echoes `hub.challenge` as plain text when `hub.mode === 'subscribe'` and `hub.verify_token === META_VERIFY_TOKEN`; returns 403 otherwise.
- `POST /webhook` — `createMetaSignatureVerifier(secrets, logger)` middleware first (`src/http/security.ts`), where `secrets` is the candidate set `[META_APP_SECRET, ...(INSTAGRAM_APP_SECRET if set)]` (the verifier accepts a match against any secret — Instagram signs with its own secret; see the load-bearing constraint below). On verified requests, the route immediately responds `200 EVENT_RECEIVED` and then calls `dispatchWebhook(req.body, logger, config)`, which (a) identifies the channel from `req.body.object`, (b) parses the payload via `parseMetaWebhook` from `src/meta/parser.ts`, (c) emits one `inbound.message` log per normalized `IncomingMessage` (warn for `type === 'unknown'`, info otherwise), one `inbound.status` log per `StatusUpdate`, and a final `inbound.{channel}` summary log with `messageCount` / `statusCount`. Channels: `'whatsapp_business_account'` → `whatsapp`, `'page'` → `messenger`, `'instagram'` → `instagram`, anything else → `unknown` (warn-logged). The parser is documented as non-throwing; the dispatcher wraps the call in a defensive `try`/`catch` and logs `dispatcher parse failed unexpectedly` as a safety net.
- `GET /health` — liveness (uptime, version from `package.json`, node version). Always mounted.

`src/meta/parser.ts` provides `parseMetaWebhook(unknown): ParseResult` plus `parseWhatsAppWebhook`, `parseMessengerWebhook`, `parseInstagramWebhook`. The normalized `IncomingMessage` / `StatusUpdate` / `ParseResult` shapes live in `src/meta/types.ts` alongside the raw per-channel envelopes. See [Message parsing](./docs/features/message-parsing.md) for the field-by-field semantics.

`scripts/` hosts the Stage 3 setup + capture tooling (separate from `src/`, which stays runtime-only):

- `scripts/lib/` — shared foundation: `tunnel.ts` (ngrok wrapper around `@ngrok/ngrok`), `graph-api.ts` (URL builder, `MetaApiError`, `graphFetch`, per-channel subscribe helpers), `console.ts` (status helpers, interactive prompts, the central `registerShutdown` registry), `capture-server.ts` (Express + tunnel + in-memory ring + signature middleware that mirrors production).
- `scripts/setup/` — `verify-shared.ts` (bootstrap, CLI parser, capture-polling predicates, result builder), `verify-{whatsapp,messenger,instagram,all}.ts` (per-channel guided verification), `oauth-instagram.ts` (Business Login OAuth → long-lived token), `register-webhooks.ts` (programmatic subscription; also reusable as a library).
- `scripts/capture/` — `fixture-capture.ts` (passive capture server with optional registration), `guided-capture.ts` (interactive scenario walker; emits self-describing wrappers to `.captures/meta/{channel}/`).

Downstream stages — per-channel outbound clients (`src/meta/{whatsapp,messenger,instagram}/client.ts`), shared Graph API helper (`src/meta/shared/graph-client.ts`), conversation agent (`src/conversation/agent.ts`), buffering (`src/conversation/buffering.ts`), conversation store and Redis-backed store (`src/conversation/{store,redis-store}.ts`), chat client (`src/chat/client.ts`), status tracker (`src/status/tracker.ts`), identity resolver (`src/identity/resolver.ts`), rate-limit tracker (`src/limits/tracker.ts`), and metrics (`src/metrics/`) — are **not present yet**. See `meta-ai-agent-implementation-plan.md` for the staged breakdown.

`src/config/loader.ts` exports `loadConfig(env = process.env)` and the `Config` type. It throws on missing `META_APP_SECRET`, `META_VERIFY_TOKEN` (< 16 chars), `CHAT_ENDPOINT_URL`, missing/invalid `NGROK_DOMAIN`, on a partially-configured channel (e.g. `WHATSAPP_PHONE_NUMBER_ID` set but `WHATSAPP_ACCESS_TOKEN` empty), and when no channel is configured at all. `INSTAGRAM_APP_SECRET` is read onto `config.instagram.appSecret` and is OPTIONAL — it does NOT gate whether the Instagram channel is "enabled" (that depends on `INSTAGRAM_USER_ID` + `INSTAGRAM_ACCESS_TOKEN`), and the loader does NOT throw when it is absent on an enabled IG channel (so WhatsApp+Messenger-only and partial setups keep running). `createApp` warns at startup if the IG channel is enabled without it, since inbound IG webhooks then fail signature verification. `META_GRAPH_API_VERSION` must match `^v\d+\.\d+$` (defaults to `v25.0`; Meta retires versions on a ~24-month cadence — bump when a newer version stabilizes).

## Load-bearing Meta platform constraints

These are the surprises that will bite future contributors. Internalize them before touching the parser, signature verifier, or outbound clients.

- **One webhook URL, three products.** A single Meta App hosts WhatsApp Cloud API, Messenger Platform, and Instagram Business Login. All three POST to the same callback URL. Distinguish by the top-level `object` field on the payload (`whatsapp_business_account`, `page`, `instagram`). One verifier serves all channels — see [Inbound webhooks](./docs/features/inbound-webhooks.md).
- **Instagram webhooks are signed with `INSTAGRAM_APP_SECRET`, NOT `META_APP_SECRET`.** WhatsApp (`object: whatsapp_business_account`) and Messenger (`object: page`) webhooks are signed with `META_APP_SECRET`; Instagram (`object: instagram`) webhooks are signed with the Instagram product's own `INSTAGRAM_APP_SECRET`. Verified against the live API 2026-05-20 — a captured IG DM webhook's HMAC matched only `INSTAGRAM_APP_SECRET`, never the Meta App secret. The signature verifier therefore tries ALL configured secrets and accepts a match against any of them (try-all, not channel-aware: verification runs on the raw bytes before JSON parsing, so we never parse untrusted input to pick a secret). `createApp` builds the candidate set from `META_APP_SECRET` plus `config.instagram.appSecret` (when set). If the Instagram channel is enabled but `INSTAGRAM_APP_SECRET` is unset, `createApp` warns at startup because inbound IG webhooks will 401. See [Webhook security](./docs/features/webhook-security.md).
- **Raw-body signature verification.** Meta signs every webhook as `X-Hub-Signature-256: sha256=HMAC_SHA256(appSecret, rawBody)`. Verification **must** run on the raw bytes Meta sent — any JSON parser that mutates whitespace, escaping, or key order will invalidate the digest. `createApp` registers `express.json({ verify: (req, _res, buf) => { req.rawBody = Buffer.from(buf); } })` for exactly this reason. See [Webhook security](./docs/features/webhook-security.md).
- **Three unrelated identifiers.** `wa_id` (WhatsApp E.164 phone), PSID (Page-Scoped ID for Messenger), and IGSID (Instagram-Scoped ID for IG DMs) are not linked by Meta. There is no API to resolve them across channels. Model identity as `(channel, channelScopedId)` tuples; cross-channel merge is an explicit app-level decision (planned: `IdentityResolver`/`ContactStore` in Stage 6).
- **WhatsApp timestamps are seconds; Messenger/Instagram are milliseconds.** The parser normalizes both forms to milliseconds at the channel boundary — WhatsApp's seconds-as-string is parsed and upscaled (`* 1000` when the numeric value is `< 1e12`). Malformed or missing timestamps fall back to `Date.now()` rather than dropping the message; losing a parseable message because of a bad timestamp is worse than logging the moment of receipt (Meta does not retry a 200-ACKed delivery). See [Message parsing](./docs/features/message-parsing.md).
- **Messenger/Instagram echoes flip `sender`/`recipient`.** For `message.is_echo === true`, Meta inverts the direction (raw `sender.id` is the BUSINESS, `recipient.id` is the USER) because the business sent the message. The parser unflips this so `channelScopedUserId` is ALWAYS the user side regardless of direction; downstream code keys on this field for conversation routing, identity resolution, and dedupe. This rule is load-bearing — do not store an echo's raw `sender.id` as the user.
- **Per-payload dedupe is parser-level only.** Both `messages[]` and `statuses[]` are deduped in-place by `channelMessageId` within a single delivery (Meta has been observed to batch identical blocks across `entry[]` items). **Cross-payload dedupe** (across Meta redeliveries on 7-day retry) is the conversation agent's responsibility (Stage 5+). Do not assume the parser handles redelivery.
- **The parser is non-throwing.** Every public entry point returns a `ParseResult` even for malformed, null, primitive, or array inputs. This is load-bearing because the handler is the dead-letter queue: a thrown parser bug would either crash the route after the 200 ACK or get swallowed and lose data. The dispatcher wraps the call in a defensive `try`/`catch` as a belt-and-suspenders safety net.
- **Reaction synthetic id is `${senderId}-${targetMessageId}-${action}` with no timestamp.** Meta retries non-2xx for 7 days, and batched events sometimes carry slightly-different timestamps for the same logical reaction. Including the timestamp in the synthetic id would break per-payload retry dedupe. Postback / referral synthetic ids deliberately keep the timestamp because they have meaningful single-payload uniqueness already.
- **Instagram does not fire webhooks for GIFs/stickers.** Per Meta's docs: "Messages with gifs and stickers are not supported. If a person sends a message with a gif or sticker a webhook will not be triggered." There is nothing to parse and no error — tolerate silently.
- **WhatsApp typing indicators require the inbound `message_id`.** A typing-on request without it is rejected (Stage 4 concern).
- **Messenger `sender_action` (`typing_on`, `mark_seen`) must be a SEPARATE request from the message.** Combining them in one POST is rejected by Meta. The Messenger client must make two calls when both are needed (Stage 4 concern).
- **Messaging windows differ per channel and matter for proactive outreach.**
  - WhatsApp: 24-hour Customer Service Window opens on each inbound. Outside the window, only pre-approved templates work (paid since 2025-07-01; marketing templates to US numbers paused since 2025-04-01). Click-to-WhatsApp ads open a 72-hour free window.
  - Messenger: 24-hour Standard Messaging Window. `HUMAN_AGENT` tag extends to 7 days but is human-only; bots cannot use it. Legacy message tags (`CONFIRMED_EVENT_UPDATE`, `ACCOUNT_UPDATE`, `POST_PURCHASE_UPDATE`) return error 100 as of 2026-04-27.
  - Instagram: 24-hour window. `HUMAN_AGENT` tag is human-only. Private Replies open a separate 7-day single-message window scoped to one comment. **There is no reliable out-of-window mechanism for an automated bot on IG today.**
- **ACK 200 immediately; process async.** Meta retries non-2xx on exponential backoff for 7 days, then permanently drops the event. There is no replay API and no dead-letter queue. The handler is the dead-letter queue. The Stage 1 POST handler already follows this rule by sending the response before dispatching.
- **WhatsApp programmatic webhook delivery requires the per-WABA `POST /{WABA_ID}/subscribed_apps` step.** The app-level `POST /{appId}/subscriptions` for `object: whatsapp_business_account` is not documented as a supported `object` value and is best-effort at best — `scripts/setup/register-webhooks.ts` skips it deliberately. Without `WHATSAPP_BUSINESS_ACCOUNT_ID` set, the script reports `manual_required` and webhooks will not actually deliver until the developer configures the Dashboard manually. With it set, the script calls `subscribeWhatsAppBusinessAccount` directly. This is the load-bearing call.
- **Instagram subscribed-field name is `messaging_referral` (singular).** Messenger uses the plural `messaging_referrals`. Both spellings live in `SUBSCRIBED_FIELDS` in `scripts/setup/register-webhooks.ts`. Mixing them up silently breaks referral webhooks on whichever channel is wrong.
- **Instagram OAuth long-lived token URL is UNVERSIONED.** The short→long token swap hits `https://graph.instagram.com/access_token?grant_type=ig_exchange_token&...` — no `/v{N.M}/` segment. `buildInstagramGraphUrl` injects a version prefix and 404s on this endpoint; `scripts/setup/oauth-instagram.ts` constructs the URL literally for this reason (see `buildLongLivedTokenUrl`).
- **Meta App credentials and Instagram product credentials are SIBLINGS, not equals.** Any Meta App that hosts the Instagram product carries two distinct credential pairs that are easy to confuse:
  - `META_APP_ID` / `META_APP_SECRET` (App Settings → Basic) — sign every webhook (`X-Hub-Signature-256` HMAC) and authorize app-level Graph API calls (e.g. `POST /{appId}/subscriptions`).
  - Instagram `client_id` (parsed from `INSTAGRAM_AUTHORIZE_URL`) / `INSTAGRAM_APP_SECRET` (Meta App Dashboard → Instagram → API setup with Instagram Business Login) — used ONLY for the Instagram OAuth code→short and short→long token exchanges.
  Confusing the two yields opaque "Invalid platform app" / "redirect_uri mismatch" errors from `api.instagram.com/oauth/access_token`. The OAuth script's startup validation enforces both env vars and never falls back to `META_APP_ID` / `META_APP_SECRET`. See [Meta setup guide](./docs/META-SETUP-GUIDE.md) and [Setup verification](./docs/features/setup-verification.md).
- **Token type per channel — System User is primarily a WhatsApp permanence concern.** Empirically validated during Stage 3 manual testing:
  - **WhatsApp Cloud API**: System User token is the required path for production-permanent tokens. The Dashboard has no equivalent "Generate Permanent Token" button for WhatsApp; user-derived tokens expire (1h short-lived, ~60d long-lived) and would break prod.
  - **Messenger**: use a **Dashboard-generated Page Access Token** (App Dashboard → Messenger → API setup → Generate Token for the Page). It is long-lived and effectively permanent so long as the issuing user retains Page admin. System User tokens are NOT accepted on the subscribed_apps endpoint — Meta's [`/{page-id}/subscribed_apps` reference](https://developers.facebook.com/docs/graph-api/reference/page/subscribed_apps/) explicitly requires "*A Page access token requested by a person who can perform CREATE_CONTENT, MANAGE, or MODERATE task on the Page*". The load-bearing phrase is "requested by a person": System User tokens are app-installed (not user-initiated) and don't satisfy that gate regardless of what scopes they carry or what Page-asset roles the System User has. The Dashboard "Generate Token" button mints a Page token from the logged-in user, which does. Symptom of the wrong token type: `subscribe_messenger_page_app failed: HTTP 403, code 210` ("Subject not visible") in `register-webhooks.ts`. `MESSENGER_PAGE_ACCESS_TOKEN` should be `type: PAGE` with `profile_id` equal to the Page id per Graph API `debug_token` — verify with `GET /debug_token?input_token=...&access_token={appId}|{appSecret}` if subscription POSTs fail.
  - **Instagram (Business Login)**: no permanent path exists. The OAuth long-lived token is ~60 days max and must be refreshed before expiry (Stage 10 will likely automate this; for now, manual rerun of `npm run setup:oauth:instagram`).
- **`MESSENGER_LOGIN_CONFIG_ID` + `scripts/setup/oauth-messenger.ts` is the canonical path to a scope-controlled Messenger Page Access Token.** App Dashboard → Messenger → "Generate Token" mints a Page token from the logged-in user's current scope grant, which is fine if those scopes already cover what you need — but it cannot add scopes the user hasn't granted (you'll get a token missing e.g. `pages_read_engagement` / `pages_manage_metadata` that 200s the audit step in `verify-messenger` but later 403s on `subscribe_messenger_page_app` with "Subject not visible"). `setup:oauth:messenger` runs the Facebook Login for Business flow against a `config_id` you create in App Dashboard → Facebook Login for Business → Configurations, which bundles the EXACT scope set server-side. Authorize URL: `https://www.facebook.com/v{N}/dialog/oauth?client_id=&config_id=&redirect_uri=&response_type=code&state=` (note: `config_id` REPLACES `scope=`; don't pass both). Code → User Token: `GET graph.facebook.com/v{N}/oauth/access_token?client_id=&client_secret=&redirect_uri=&code=`. User Token → Page Token: `GET /me/accounts` returns one entry per managed Page, each with its own `access_token` (that's the Page token). Page tokens derived from long-lived User tokens are effectively non-expiring as long as the issuing user retains Page admin.
- **`INSTAGRAM_AUTHORIZE_URL` is the single source of truth for the Instagram OAuth redirect URI and client_id.** The user pastes the embed authorize URL from Meta App Dashboard → Instagram → API setup with Instagram Business Login → "Authorize this app for Instagram business"; `oauth-instagram.ts` parses `client_id`, `redirect_uri`, and (when present) `state` out of it via `parseAuthorizeUrl`, and never constructs an authorize URL itself in the live flow. Meta's Dashboard embed URL omits `state` — when absent, `generateState()` produces a fresh 16-byte hex nonce and `withState()` appends it to the URL before opening the browser. Earlier CLI overrides for the redirect URI were removed when the env var became the only input; only `--reveal` and `--help` remain. The unused `buildAuthorizeUrl` / `IG_SCOPES` / `validateRedirectUri` helpers were removed in the post-Stage-3 dead-code sweep — the script no longer constructs authorize URLs or scope lists internally.
- **`registerShutdown` from `scripts/lib/console.ts` is the official cleanup-hook surface.** Do not install your own `process.on('SIGINT')` in new setup/capture scripts. The shared registry runs all hooks in parallel with a 5-second budget and avoids races between readline cleanup and per-script handles (tunnel, server). Earlier code had competing handlers; the registry pattern is the resolution.
- **`NGROK_DOMAIN` is required (no longer optional).** Three Meta Dashboard webhook callback URL registrations (WhatsApp, Messenger, Instagram) plus the Instagram Business Login OAuth redirect URI are all anchored to a single public domain. Without a pinned static domain (free tier includes one at https://dashboard.ngrok.com/cloud-edge/domains), every run forces re-registering all four entries in the Dashboard — there is no Graph API to atomically update a registered callback URL. `loadConfig` validates that the value is a bare hostname (no scheme, no path, contains a `.`) and throws otherwise. `TunnelOptions.domain` is also required across `scripts/lib/tunnel.ts` callers; all setup/capture scripts read it from `loadConfig().ngrokDomain` (or accept a `--ngrok-domain=<x>` CLI override).
- **ngrok free-tier static domains end in `.ngrok-free.dev`; paid-tier in `.ngrok.app` / `.ngrok-free.app`.** `NGROK_DOMAIN` is the bare hostname (no `https://`, no path) — the ngrok SDK adds the scheme. Don't copy `.ngrok-free.app` placeholders from older docs verbatim; the ngrok Dashboard at https://dashboard.ngrok.com/cloud-edge/domains is the source of truth for your actual reserved domain.

## Conventions

- TypeScript ESM (`"type": "module"`). Internal imports of `.ts` files use `.js` specifiers (Node ESM resolution: `import { createApp } from './http/app.js'`).
- Node `>=20`. Vitest for tests; `node-mocks-http` for Express handler unit tests; `supertest` for integration tests against the real Express app.
- Structured logging with `pino`. Include enough fields to debug webhook delivery, signature failures, channel routing, and (later) chat-endpoint failures. The integration tests assert specific `traceMarker` values — `inbound.whatsapp`, `inbound.messenger`, `inbound.instagram`, `inbound.unknown` — preserve those when extending the dispatcher.
- Capture artifacts (`.captures/`) and `.env` are gitignored and may contain phone numbers, tunnel URLs, message content, and secrets — manually redact before promoting captures into `tests/fixtures/meta/captured/` (Stage 3 workflow — see [Payload capture](./docs/features/payload-capture.md)).
- Stage 3 setup and capture scripts are run via `node --import tsx scripts/...` (see `package.json` scripts). Each script calls `import 'dotenv/config'` at the top to load `.env`, then `loadConfig()` from `src/config/loader.ts`. The `oauth-instagram.ts` script reads `INSTAGRAM_AUTHORIZE_URL`, `INSTAGRAM_APP_SECRET`, and `NGROK_DOMAIN` directly (bypassing `loadConfig`) because it doesn't need `META_VERIFY_TOKEN` or `CHAT_ENDPOINT_URL` to run. It deliberately does NOT consume `META_APP_ID` / `META_APP_SECRET` — Instagram product OAuth uses its own credential pair (see the load-bearing constraint on Meta vs. Instagram credentials). `oauth-messenger.ts` likewise reads env directly, but DOES consume `META_APP_ID` / `META_APP_SECRET` (the Messenger OAuth flow authenticates as the Meta App itself) plus `MESSENGER_LOGIN_CONFIG_ID` and `NGROK_DOMAIN`.
- New feature docs follow `docs/features/<name>.md`: what it does, how it works, code files, configuration, known limitations.
