# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Scope

`meta-ai-agent` is a transport/orchestration package — Meta webhook ingestion, signature verification, outbound delivery, status tracking, dedupe, conversation buffering, and ordered delivery for an AI agent that runs over WhatsApp Cloud API, Facebook Messenger, and Instagram DMs under a single Meta App. The developer brings their own HTTP chat endpoint. The transport package itself stays model-provider-free: do not couple `src/` to a model provider, add LLM-specific assumptions to the chat contract, or pull OpenAI/Anthropic/etc. SDKs into the root `package.json`.

`meta-ai-agent-implementation-plan.md` is the durable project guidance (intent, ten-stage roadmap, Meta platform constraints, testing strategy, implementation rules). Read it before non-trivial changes — it lists API quirks (single-app/three-products webhook fan-in, channel-specific timestamp/typing/echo rules, messaging windows, retry behavior, token lifecycle) that the code intentionally relies on.

## Stage X status

**Stages 1–6 complete: scaffolding, signature verification, webhook routing, payload parsing, normalized message types, setup verification, payload capture, outbound send adapters (text, typing, read receipts, reactions, WhatsApp templates) across all three channels via a shared runtime `GraphClient` and a common `ChannelAdapter` interface, the conversation agent (inbound burst buffering, the chat-endpoint contract, ordered channel-aware outbound delivery, an in-memory conversation store with dedupe, and per-conversation serialization), + status tracking, optional identity enrichment, and operational visibility (metrics, per-request tracing, health/ready/metrics/admin routes with PII-redacted output). Stages 7–10 planned — see `meta-ai-agent-implementation-plan.md`.** Stages 1–5 are the critical path; Stage 6 is the operational surface on top. No persistence (Redis/BullMQ), no media send, no full rate limiting yet — and the metrics/status/contact stores are in-memory until the Redis swap in Stage 10. The POST `/webhook` handler verifies the Meta signature, ACKs 200, parses the payload via `parseMetaWebhook`, emits per-message / per-status / per-channel structured logs (now under a request-scoped trace id + child logger), counts webhook metrics, and (when an agent is wired) routes each parsed message/status into the `ConversationAgent`, which buffers the inbound burst, resolves an optional contact (fail-open), marks the inbound message read when `READ_RECEIPTS_ENABLED` (at flush, before the chat call — so silent and reaction-only turns still mark read), calls the developer's chat endpoint, drives ordered outbound delivery through the `ChannelAdapter`s, and records delivery-status history. See [Conversation state](./docs/features/conversation-state.md), [Message buffering](./docs/features/message-buffering.md), [Ordered delivery](./docs/features/ordered-delivery.md), [Rich chat actions](./docs/features/rich-chat-actions.md), [Outbound clients](./docs/features/outbound-clients.md), [Status tracking](./docs/features/status-tracking.md), [Read receipts](./docs/features/read-receipts.md), [Identity resolution](./docs/features/identity-resolution.md), and [Operational visibility](./docs/features/operational-visibility.md).

## Architecture

The runtime is a single Express app composed in `src/http/app.ts`. `createApp({ config, logger })` wires:

- `express.json({ limit: '5mb', verify })` first. The `verify` hook copies the raw body buffer onto `req.rawBody` **before** JSON parsing mutates it — signature verification depends on byte-exact body capture. This is load-bearing; do not introduce another body parser ahead of it without preserving `rawBody`.
- `GET /webhook` — Meta verification handshake. Echoes `hub.challenge` as plain text when `hub.mode === 'subscribe'` and `hub.verify_token === META_VERIFY_TOKEN`; returns 403 otherwise.
- `POST /webhook` — `createMetaSignatureVerifier(secrets, logger)` middleware first (`src/http/security.ts`), where `secrets` is the candidate set `[META_APP_SECRET, ...(INSTAGRAM_APP_SECRET if set)]` (the verifier accepts a match against any secret — Instagram signs with its own secret; see the load-bearing constraint below). On verified requests, the route immediately responds `200 EVENT_RECEIVED` and then calls `dispatchWebhook(req.body, logger, config, agent)`, which (a) identifies the channel from `req.body.object`, (b) parses the payload via `parseMetaWebhook` from `src/meta/parser.ts`, (c) emits one `inbound.message` log per normalized `IncomingMessage` (warn for `type === 'unknown'`, info otherwise), one `inbound.status` log per `StatusUpdate`, and a final `inbound.{channel}` summary log with `messageCount` / `statusCount`, and (d) when an `agent` is supplied, routes each parsed message into `agent.handleInbound` and each status into `agent.handleStatus` in a sequential awaited loop AFTER the logs. The agent is OPTIONAL on `createApp`: the parse+log shapes are unchanged whether or not it is wired (the parse+log-only tests construct the app without an agent). The sequential await preserves intra-webhook arrival order (a webhook routinely batches several messages for one conversation); it is still fire-and-forget from the route (the 200 is already sent) and every `handle*` is fail-soft, so the loop cannot reject. Channels: `'whatsapp_business_account'` → `whatsapp`, `'page'` → `messenger`, `'instagram'` → `instagram`, anything else → `unknown` (warn-logged). The parser is documented as non-throwing; the dispatcher wraps the call in a defensive `try`/`catch` and logs `dispatcher parse failed unexpectedly` as a safety net.
- `GET /health` — liveness (uptime, version from `package.json`, node version). Always mounted, unauthenticated.
- `GET /ready` — readiness (Stage 6). Always mounted, unauthenticated. `buildReadinessReport` runs each dependency check defensively (a thrown check fails THAT check, not the route) and returns 503 if any fails: `checks.scheduler` (via `scheduler.getStats()`) and `checks.redis` (presence-only — `configured` vs `not_configured`; the real Redis ping is Stage 10).
- `GET /metrics` — Prometheus exposition (Stage 6), token-gated. **Guarded at registration**: mounted only when `ADMIN_API_TOKEN` is set AND a `metricsCollector` is wired; unmounted → 404 (never 401).
- `GET /admin/conversations/:key` and `GET /admin/status/:messageId` — token-gated admin introspection (Stage 6), **PII-redacted by default** (allow-list / fail-closed; `?reveal=true` unmasks). Same registration guard. Auth (`src/http/auth.ts`) is constant-time over `Authorization: Bearer` or `x-admin-api-token`.
- A `traceMiddleware` (`src/http/trace.ts`) is mounted after `express.json` and before the routes: it validates/echoes an `x-trace-id` (CRLF/injection-guarded against `^[A-Za-z0-9._:-]{1,128}$`, else a fresh uuid) and puts a pino child logger on `res.locals`; the webhook dispatcher threads `{traceId, logger}` into the agent.

`src/meta/parser.ts` provides `parseMetaWebhook(unknown): ParseResult` plus `parseWhatsAppWebhook`, `parseMessengerWebhook`, `parseInstagramWebhook`. The normalized `IncomingMessage` / `StatusUpdate` / `ParseResult` shapes live in `src/meta/types.ts` alongside the raw per-channel envelopes. See [Message parsing](./docs/features/message-parsing.md) for the field-by-field semantics.

`src/meta/shared/` holds the Stage 4 outbound foundation: `errors.ts` (the canonical `MetaApiError` — see the runtime-vs-setup split below), `graph-client.ts` (`GraphClient`, the runtime HTTP transport with retry/backoff and Bearer-header auth; transport-only, no channel body shapes), and `adapter.ts` (the `ChannelAdapter` interface plus `SendResult` / `SendOptions` / `ChannelFeature`). The three per-channel send clients — `src/meta/whatsapp/client.ts`, `src/meta/messenger/client.ts`, `src/meta/instagram/client.ts` — each implement `ChannelAdapter` on top of an injected `GraphClient`, owning only their channel-specific request bodies (WhatsApp `messaging_product`/template/typing-read coupling; Messenger/Instagram `recipient`/`sender_action`). **Runtime-vs-setup client split:** the runtime `GraphClient` (`src/meta/shared/graph-client.ts`) is SEPARATE from the setup-time `graphFetch` (`scripts/lib/graph-api.ts`); `MetaApiError` is canonical in `src/meta/shared/errors.ts` and re-exported by `scripts/lib/graph-api.ts` (one error class). `src/` never imports from `scripts/` — the dependency direction is one-way (`scripts/ → src/`). See [Outbound clients](./docs/features/outbound-clients.md).

`scripts/` hosts the Stage 3 setup + capture tooling (separate from `src/`, which stays runtime-only):

- `scripts/lib/` — shared foundation: `tunnel.ts` (ngrok wrapper around `@ngrok/ngrok`), `graph-api.ts` (URL builder, `MetaApiError`, `graphFetch`, per-channel subscribe helpers), `console.ts` (status helpers, interactive prompts, the central `registerShutdown` registry), `capture-server.ts` (Express + tunnel + in-memory ring + signature middleware that mirrors production).
- `scripts/setup/` — `verify-shared.ts` (bootstrap, CLI parser, capture-polling predicates, result builder), `verify-{whatsapp,messenger,instagram,all}.ts` (per-channel guided verification), `oauth-instagram.ts` (Business Login OAuth → long-lived token), `register-webhooks.ts` (programmatic subscription; also reusable as a library).
- `scripts/capture/` — `fixture-capture.ts` (passive capture server with optional registration), `guided-capture.ts` (interactive scenario walker; emits self-describing wrappers to `.captures/meta/{channel}/`).

`src/conversation/`, `src/chat/`, and `src/delivery/` hold the Stage 5 conversation layer. `src/conversation/agent.ts` (`ConversationAgent`) is the central state machine: `idle → buffering → processing → sending → idle`. It owns one record per (channel, business, user) triple, buffers inbound bursts (`src/conversation/buffering.ts` + `scheduler.ts`), calls the developer's chat endpoint (`src/chat/client.ts` + `contract.ts`), and drives an ordered, channel-aware outbound queue (`src/delivery/queue.ts`). State + dedupe + outbound-handle mapping live in `src/conversation/store.ts` (`InMemoryConversationStore`). `src/index.ts` exports `buildRuntime(config, logger): { app, agent }`, which builds the full graph: a shared `GraphClient`, one `ChannelAdapter` per configured channel, an `InMemoryConversationStore`, an `InMemoryBufferScheduler`, an `HttpChatClient`, the Stage 6 observability deps (`InMemoryMetricsCollector` + `createAgentMetrics` — `agentUp`/`agentBuildInfo` set immediately; an `HttpIdentityResolver` over `USER_LOOKUP_URL` or a `NoopIdentityResolver`; an `InMemoryStatusTracker`), and the `ConversationAgent`, then `createApp({ config, logger, agent, metrics, metricsCollector, statusTracker, store, scheduler })`. All the Stage 6 deps on `createApp` are optional, so the parse+log-only and Stage 5 callers keep compiling. `main()` calls `buildRuntime` and starts the listener; the `dev:loop` harness (`scripts/dev/loop.ts`) reuses `buildRuntime` to boot the exact same runtime against an alternate chat endpoint + tunnel. `main()`'s autostart is guarded by an `invokedAsScript` check, so importing `buildRuntime` does NOT start a server. See [Conversation state](./docs/features/conversation-state.md), [Message buffering](./docs/features/message-buffering.md), [Ordered delivery](./docs/features/ordered-delivery.md), and [Rich chat actions](./docs/features/rich-chat-actions.md).

`src/status/`, `src/identity/`, `src/metrics/`, and the Stage 6 `src/http/` primitives hold the operational surface. `src/status/{types,tracker}.ts` is the delivery-status tracker (`InMemoryStatusTracker`: per-outbound-message `StatusRecord` with a rank-based non-regressing `current`, append-only `history`, idempotent on `(status,timestamp)`; WhatsApp records per-message via `applyStatusUpdate`, Messenger/IG via `applyReadWatermark` after the agent translates the read watermark to message ids). `src/identity/{resolver,contact-store}.ts` is optional, fail-open identity enrichment over `USER_LOOKUP_URL` (`HttpIdentityResolver` cache-then-fetch into `InMemoryContactStore`, resolve-once-per-conversation; `NoopIdentityResolver` when the URL is unset). `src/metrics/{collector,registry,prometheus}.ts` is the provider-agnostic metrics layer (`MetricsCollector` interface, `InMemoryMetricsCollector` with a per-metric cardinality cap + `NoopMetricsCollector`, `createAgentMetrics` named handles, `renderPrometheus` text exposition). `src/http/{trace,auth,redaction}.ts` are the trace middleware, the constant-time admin-token check, and the allow-list/fail-closed PII redactors. The conversation agent threads all three in (metrics on every path, fail-open identity in `handleInboundImpl`, status history in `handleStatusImpl` + `handleReadWatermarkImpl`). See [Status tracking](./docs/features/status-tracking.md), [Read receipts](./docs/features/read-receipts.md), [Identity resolution](./docs/features/identity-resolution.md), and [Operational visibility](./docs/features/operational-visibility.md). Still **not present yet**: the Redis-backed store + BullMQ scheduler (`src/conversation/redis-store.ts` and the `'bullmq'` scheduler, selected on `REDIS_URL`), the rate-limit tracker (`src/limits/tracker.ts`), and media send (`src/meta/shared/media.ts`). The Stage 6 metrics/status/contact stores are all in-memory until the Redis swap in Stage 10. See `meta-ai-agent-implementation-plan.md` for the staged breakdown.

`src/config/loader.ts` exports `loadConfig(env = process.env)` and the `Config` type. It throws on missing `META_APP_SECRET`, `META_VERIFY_TOKEN` (< 16 chars), `CHAT_ENDPOINT_URL`, missing/invalid `NGROK_DOMAIN`, an invalid `USER_LOOKUP_URL` (optional, but when set must parse as a URL), an `ADMIN_API_TOKEN` shorter than 16 chars (optional, but ≥16 when set — ≥32 recommended; it gates the PII-bearing admin/metrics routes), on a partially-configured channel (e.g. `WHATSAPP_PHONE_NUMBER_ID` set but `WHATSAPP_ACCESS_TOKEN` empty), and when no channel is configured at all. `USER_LOOKUP_TIMEOUT_MS` (default 5000) lands in the nested `conversation` section. `INSTAGRAM_APP_SECRET` is read onto `config.instagram.appSecret` and is OPTIONAL — it does NOT gate whether the Instagram channel is "enabled" (that depends on `INSTAGRAM_USER_ID` + `INSTAGRAM_ACCESS_TOKEN`), and the loader does NOT throw when it is absent on an enabled IG channel (so WhatsApp+Messenger-only and partial setups keep running). `createApp` warns at startup if the IG channel is enabled without it, since inbound IG webhooks then fail signature verification. `META_GRAPH_API_VERSION` must match `^v\d+\.\d+$` (defaults to `v25.0`; Meta retires versions on a ~24-month cadence — bump when a newer version stabilizes).

## Load-bearing Meta platform constraints

These are the surprises that will bite future contributors. Internalize them before touching the parser, signature verifier, or outbound clients.

- **One webhook URL, three products.** A single Meta App hosts WhatsApp Cloud API, Messenger Platform, and Instagram Business Login. All three POST to the same callback URL. Distinguish by the top-level `object` field on the payload (`whatsapp_business_account`, `page`, `instagram`). One verifier serves all channels — see [Inbound webhooks](./docs/features/inbound-webhooks.md).
- **Instagram webhooks are signed with `INSTAGRAM_APP_SECRET`, NOT `META_APP_SECRET`.** WhatsApp (`object: whatsapp_business_account`) and Messenger (`object: page`) webhooks are signed with `META_APP_SECRET`; Instagram (`object: instagram`) webhooks are signed with the Instagram product's own `INSTAGRAM_APP_SECRET`. Verified against the live API 2026-05-20 — a captured IG DM webhook's HMAC matched only `INSTAGRAM_APP_SECRET`, never the Meta App secret. The signature verifier therefore tries ALL configured secrets and accepts a match against any of them (try-all, not channel-aware: verification runs on the raw bytes before JSON parsing, so we never parse untrusted input to pick a secret). `createApp` builds the candidate set from `META_APP_SECRET` plus `config.instagram.appSecret` (when set). If the Instagram channel is enabled but `INSTAGRAM_APP_SECRET` is unset, `createApp` warns at startup because inbound IG webhooks will 401. See [Webhook security](./docs/features/webhook-security.md).
- **Raw-body signature verification.** Meta signs every webhook as `X-Hub-Signature-256: sha256=HMAC_SHA256(appSecret, rawBody)`. Verification **must** run on the raw bytes Meta sent — any JSON parser that mutates whitespace, escaping, or key order will invalidate the digest. `createApp` registers `express.json({ verify: (req, _res, buf) => { req.rawBody = Buffer.from(buf); } })` for exactly this reason. See [Webhook security](./docs/features/webhook-security.md).
- **Three unrelated identifiers.** `wa_id` (WhatsApp E.164 phone), PSID (Page-Scoped ID for Messenger), and IGSID (Instagram-Scoped ID for IG DMs) are not linked by Meta. There is no API to resolve them across channels. Model identity as `(channel, channelScopedId)` tuples; cross-channel merge is an explicit app-level decision (planned: `IdentityResolver`/`ContactStore` in Stage 6).
- **WhatsApp timestamps are seconds; Messenger/Instagram are milliseconds.** The parser normalizes both forms to milliseconds at the channel boundary — WhatsApp's seconds-as-string is parsed and upscaled (`* 1000` when the numeric value is `< 1e12`). Malformed or missing timestamps fall back to `Date.now()` rather than dropping the message; losing a parseable message because of a bad timestamp is worse than logging the moment of receipt (Meta does not retry a 200-ACKed delivery). See [Message parsing](./docs/features/message-parsing.md).
- **Messenger/Instagram echoes flip `sender`/`recipient`.** For `message.is_echo === true`, Meta inverts the direction (raw `sender.id` is the BUSINESS, `recipient.id` is the USER) because the business sent the message. The parser unflips this so `channelScopedUserId` is ALWAYS the user side regardless of direction; downstream code keys on this field for conversation routing, identity resolution, and dedupe. This rule is load-bearing — do not store an echo's raw `sender.id` as the user.
- **Per-payload dedupe is parser-level only.** Both `messages[]` and `statuses[]` are deduped in-place by `channelMessageId` within a single delivery (Meta has been observed to batch identical blocks across `entry[]` items). **Cross-payload dedupe** (across Meta redeliveries on 7-day retry) is the conversation agent's responsibility (Stage 5+). Do not assume the parser handles redelivery.
- **The parser is non-throwing.** Every public entry point returns a `ParseResult` even for malformed, null, primitive, or array inputs. This is load-bearing because the handler is the dead-letter queue: a thrown parser bug would either crash the route after the 200 ACK or get swallowed and lose data. The dispatcher wraps the call in a defensive `try`/`catch` as a belt-and-suspenders safety net.
- **Reaction synthetic id is `${senderId}-${targetMessageId}-${action}` with no timestamp.** Meta retries non-2xx for 7 days, and batched events sometimes carry slightly-different timestamps for the same logical reaction. Including the timestamp in the synthetic id would break per-payload retry dedupe. Postback / referral synthetic ids deliberately keep the timestamp because they have meaningful single-payload uniqueness already.
- **Instagram does not fire webhooks for GIFs/stickers.** Per Meta's docs: "Messages with gifs and stickers are not supported. If a person sends a message with a gif or sticker a webhook will not be triggered." There is nothing to parse and no error — tolerate silently.
- **Outbound POST 5xx is NOT retried — double-send safety.** The runtime `GraphClient` (`src/meta/shared/graph-client.ts`) retries `429` and pre-response network failures (`httpStatus 0`) on ANY method, but retries `5xx` ONLY for idempotent requests. All sends are POST and leave `idempotent` unset, so a 5xx after a send is NOT retried: a 5xx may have surfaced after Meta already accepted and delivered the message, and retrying could double-send. Do not flip `idempotent: true` on a send to "make it more reliable." Backoff honors `Retry-After`, else exponential + jitter. See [Outbound clients](./docs/features/outbound-clients.md).
- **Outbound auth is `Authorization: Bearer <token>`, never the query string.** Tokens in URLs leak into proxy/CDN/access logs; the runtime client always sends them as a header. Same precedent as the webhook surface.
- **WhatsApp typing is a COMBINED call coupled with mark-read and requires the inbound `message_id`.** There is no standalone WhatsApp "typing on". `sendTypingIndicator` POSTs `{ messaging_product:'whatsapp', status:'read', message_id:<inbound wamid>, typing_indicator:{ type:'text' } }` — it marks that inbound message read AND attaches the typing bubble in one request. Without an inbound `message_id` there is nothing to mark read and no way to trigger typing; the client logs a warn and skips. (The implementation plan's older description of a standalone `type:'typing_indicator'` message is INCORRECT — the code uses this combined call; see [Known gaps](./docs/KNOWN-GAPS.md).)
- **Messenger/Instagram `sender_action` (`typing_on`/`typing_off`, `mark_seen`, `react`/`unreact`) must be a SEPARATE request from a message.** Meta rejects a body that combines a `sender_action` with a `message`. The client makes two POSTs when both are needed (sequencing is the conversation agent's job in Stage 5).
- **Reactions are supported outbound on all three channels, with two body shapes.** WhatsApp: `type:'reaction'`, `reaction:{ message_id, emoji }` (empty `emoji` = unreact, preserved verbatim). Messenger & Instagram: `sender_action:'react'` with `payload:{ message_id, reaction:emoji }` (emoji nested inside `payload`, not a sibling); unreact = `sender_action:'unreact'` with `payload:{ message_id }`. WhatsApp's reaction send requires `to` in the body; Messenger/IG derive the target from `message_id`.
- **Quoted replies use a different mechanism per channel; Instagram-Login has none.** WhatsApp threads a reply via `context:{ message_id }`; Messenger via a **top-level** `reply_to:{ mid }` (a sibling of `message`, NOT `message.reply_to` — both live-confirmed working 2026-05-20). Instagram (Instagram-Login, `graph.instagram.com`) has **no working outbound quoted reply**: exhaustively verified 2026-05-20, every `reply_to` shape/target returns `code 100 / subcode 2534002 "Invalid Message ID"` (even for a bot's own just-returned valid mid) or is silently ignored. So `InstagramClient.supports('reply_to')` is `false`, `sendText` builds no reply field, and the delivery queue (`buildOutboundItems`) downgrades a `reply` action to a plain `message` so the user still gets the text. Do not re-add IG `reply_to`. See [Outbound clients](./docs/features/outbound-clients.md) and [Known gaps](./docs/KNOWN-GAPS.md).
- **Instagram messaging uses host `graph.instagram.com`, not `graph.facebook.com`.** Every IG send (`POST {igUserId}/messages`) targets that host (Business Login messaging endpoints). IG sends also pass through a minimal in-process pacer (default 100ms min inter-call interval, constructor-overridable) — a conservative floor only; the real ~300/sec text + ~10/sec media + hourly throughput model is deferred to Stage 10's `LimitTracker`.
- **Messaging windows differ per channel and matter for proactive outreach.**
  - WhatsApp: 24-hour Customer Service Window opens on each inbound. Outside the window, only pre-approved templates work (paid since 2025-07-01; marketing templates to US numbers paused since 2025-04-01). Click-to-WhatsApp ads open a 72-hour free window.
  - Messenger: 24-hour Standard Messaging Window. `HUMAN_AGENT` tag extends to 7 days but is human-only; bots cannot use it. Legacy message tags (`CONFIRMED_EVENT_UPDATE`, `ACCOUNT_UPDATE`, `POST_PURCHASE_UPDATE`) return error 100 as of 2026-04-27.
  - Instagram: 24-hour window. `HUMAN_AGENT` tag is human-only. Private Replies open a separate 7-day single-message window scoped to one comment. **There is no reliable out-of-window mechanism for an automated bot on IG today.**
- **ACK 200 immediately; process async.** Meta retries non-2xx on exponential backoff for 7 days, then permanently drops the event. There is no replay API and no dead-letter queue. The handler is the dead-letter queue. The Stage 1 POST handler already follows this rule by sending the response before dispatching.

## Load-bearing conversation-agent constraints (Stage 5)

These govern `src/conversation/agent.ts` and the store/scheduler. Internalize them before editing the agent — several encode confirmed-bug fixes. See [Conversation state](./docs/features/conversation-state.md).

- **Per-conversation serialization lock — entry-point vs internal split.** The store is pass-by-value with last-write-wins (no atomic read-modify-write); the in-memory store clones on read/write but that does NOT prevent a lost update. Two concurrent flows for one key both read the same clone, mutate, and write — the second clobbers the first and a user message (or queue advance) is silently lost. This was a CONFIRMED message-dropping race (the dispatcher fires inbounds/statuses concurrently and a single webhook batches many messages for one conversation). `runExclusive(key, fn)` serializes per key via a promise tail; different keys still run concurrently. The split is load-bearing: **entry points ACQUIRE** (`handleInbound`, `handleStatus`, the scheduler flush handler, the delivery-timeout callback — each fires outside any held lock), **internal `*Impl`/helpers stay LOCK-FREE** (they assume the lock is held). `handleReaction` delegates to `handleInbound` and must NOT acquire (it would deadlock). No lock-holding path may call a same-key acquirer. Preserve this when adding methods.
- **Channel-aware queue advancement + delivery-timeout fallback.** WhatsApp advances the outbound queue only on a `sent`/`delivered` status webhook (`on_status`, `handleStatus`); Messenger/Instagram advance the moment a send returns (`on_send`) because they have no reliable per-message delivery callback. `advancementMode`/`statusAdvancesQueue` in `src/delivery/queue.ts` encode this. WhatsApp arms an `OUTBOUND_DELIVERY_TIMEOUT_MS` fallback so a dropped status can't wedge the queue; the clear-before-arm (one timer per key) plus the `currentOutboundIndex === messageIndex` guard make it safe from double-advancing. Reaction/typing items always advance-on-send (no channel emits a status for them). Outbound message ids are mapped back to conversation+index (`mapOutboundHandle`) so a status — which carries only the channel message id — correlates. See [Ordered delivery](./docs/features/ordered-delivery.md).
- **Buffer timeout must stay > 0 (no inline scheduler fire).** `InMemoryBufferScheduler.schedule` fires the handler INLINE (synchronously) when `delayMs <= 0`. `handleInboundImpl` calls `schedule` WHILE holding the per-key lock, and the flush handler re-acquires that same key's lock — so an inline fire self-deadlocks. `calculateBufferTimeout` never returns `<= 0` for a valid config (`bufferBaseTimeoutMs` is a positive int; jitter is clamped to `>= base*0.5`), so `schedule` always takes the `setTimeout` branch and the handler fires later, outside the lock. Keep this invariant if the buffer math changes. See [Message buffering](./docs/features/message-buffering.md).
- **Echo-filter THEN dedupe, in that order.** `handleInboundImpl` drops `isEcho` messages BEFORE the `claimInboundHandle` (SETNX) dedupe claim. Meta echoes business-sent messages back (Messenger/IG `is_echo`); treating an echo as inbound would loop the agent's own output back into the chat endpoint, and filtering first means an echo never consumes a dedupe slot. Cross-payload dedupe is `store.claimInboundHandle(channelMessageId)` with TTL `DEDUPE_TTL_SECONDS` (distinct from the parser's per-payload dedupe).
- **`handle*` are fail-soft — never throw out.** The HTTP layer already ACKed 200, so `handleInbound`/`handleStatus`/the flush body catch, log, and return. Chat-endpoint and adapter send errors are logged; a chat error ends the turn quietly, a single bad send is marked skipped and the queue advances. A thrown error here would crash the route after the ACK or get swallowed and lose data.
- **In-memory store + scheduler now; Redis/BullMQ in Stage 10.** `InMemoryConversationStore` + `InMemoryBufferScheduler` are per-process and lost on restart (the per-replica view diverges in a multi-replica deploy). The production swap to a Redis-backed store (real `SET NX`, `SCAN`) + BullMQ scheduler is selected on `REDIS_URL` and lands in Stage 10. The in-memory dedupe map is never swept — it relies on the later Redis TTL. The `ConversationStore`/`BufferScheduler` interfaces are the contract both impls honor.

## Load-bearing operational-visibility constraints (Stage 6)

These govern `src/{status,metrics,identity}/` and the Stage 6 `src/http/` primitives. Internalize them before touching the admin surface, metrics, or the agent's Stage 6 wiring. See [Operational visibility](./docs/features/operational-visibility.md).

- **Admin + `/metrics` routes are guarded AT REGISTRATION (404 when token unset, not 401).** `GET /metrics`, `GET /admin/conversations/:key`, and `GET /admin/status/:messageId` are wrapped in `if (config.adminApiToken && <dep>) { app.get(...) }` in `src/http/app.ts`. When `ADMIN_API_TOKEN` is unset the handler is never registered, so a request 404s through the catch-all — we never advertise an admin surface on a deploy that hasn't configured a token. When the token IS set, an absent/wrong token 401s. `/health` and `/ready` are the ONLY always-on, unauthenticated routes. Do not "simplify" this to an always-mounted route that 401s.
- **Admin output is allow-list redacted (fail-CLOSED) — new PII fields must be explicitly allow-listed.** The inbound/outbound/record/status redactors in `src/http/redaction.ts` build a NEW object copying only allow-listed structural fields, mask the known content fields, and DROP everything else. A field added to `IncomingMessage`/`OutboundItem`/`ConversationRecord`/`StatusRecord` later is OMITTED from the masked view until someone allow-lists it — so new PII fails closed. Do NOT switch any of these to a spread-then-deny-list. (`redactContact` is the one intentional spread: `tags`/`customVariables`/`unifiedContactId` are developer-supplied operational metadata, kept by design — a recorded gap.) `?reveal=true` (authenticated only) returns the source untouched.
- **Identity enrichment is FAIL-OPEN.** `HttpIdentityResolver.resolve` NEVER throws — a missing/non-2xx/network/timeout/parse failure all resolve to `undefined` ("proceed without enrichment"). The agent treats `undefined` uniformly and the conversation continues. A misbehaving `USER_LOOKUP_URL` must never stall or break message delivery. Logging is PII-safe (boolean `enriched` + redacted last-4 id only).
- **Status watermark reads are observability-ONLY (no queue effect).** Messenger/IG read events are a watermark, not a per-message id; the agent translates them to concrete sent ids and marks them `read` in the tracker (`handleReadWatermarkImpl`). Because Messenger/IG are advance-on-send, the queue already moved past every sent item — a read must NEVER advance, skip, or re-open a queue item. WhatsApp's queue advances on `sent`/`delivered`, never on `read`. Keep the watermark path side-effect-free on the queue.
- **`x-trace-id` is validated against injection.** `traceMiddleware` accepts an inbound `x-trace-id` only if it matches `^[A-Za-z0-9._:-]{1,128}$`, else it mints a `randomUUID`. The id is echoed in a response header AND stamped into log lines / the pino child's bindings, so reflecting untrusted bytes would enable CRLF / header / log injection. Never relax the charset or skip the validation.
- **Metric labels must stay bounded — no raw ids or message text.** Labels are bounded enums (`channel`/`type`/`result`/`operation`/`status`) plus `error_code` normalized via `normalizeErrorCodeLabel` (unknown → `other`, missing → `none`). `InMemoryMetricsCollector` also caps distinct label combinations per metric (folding overflow into `__overflow__`). Never add a label derived from a user id, phone number, or message body — it would explode cardinality and leak PII into the metrics surface.
- **`ADMIN_API_TOKEN` ≥16 chars.** When set it must be ≥16 (≥32 recommended); `loadConfig` throws otherwise. It guards PII-bearing routes, so a trivially short bearer token is a foot-gun we fail fast on at boot.
- **In-memory metrics / status / contact stores now; Redis in Stage 10.** `InMemoryMetricsCollector`, `InMemoryStatusTracker`, and `InMemoryContactStore` are per-process, lost on restart, and (apart from the metric cardinality cap) unbounded with no TTL/sweeper — acceptable because the Redis-backed impls with TTL eviction land in Stage 10. The collector/tracker/store interfaces are the contract those impls will honor.

- **WhatsApp programmatic webhook delivery requires the per-WABA `POST /{WABA_ID}/subscribed_apps` step.** The app-level `POST /{appId}/subscriptions` for `object: whatsapp_business_account` is not documented as a supported `object` value and is best-effort at best — `scripts/setup/register-webhooks.ts` skips it deliberately. Without `WHATSAPP_BUSINESS_ACCOUNT_ID` set, the script reports `manual_required` and webhooks will not actually deliver until the developer configures the Dashboard manually. With it set, the script calls `subscribeWhatsAppBusinessAccount` directly. This is the load-bearing call.
- **Instagram subscribed-field name is `messaging_referral` (singular).** Messenger uses the plural `messaging_referrals`. Both spellings live in `SUBSCRIBED_FIELDS` in `scripts/setup/register-webhooks.ts`. Mixing them up silently breaks referral webhooks on whichever channel is wrong.
- **Instagram OAuth long-lived token URL is UNVERSIONED.** The short→long token swap hits `https://graph.instagram.com/access_token?grant_type=ig_exchange_token&...` — no `/v{N.M}/` segment. `buildInstagramGraphUrl` injects a version prefix and 404s on this endpoint; `scripts/setup/oauth-instagram.ts` constructs the URL literally for this reason (see `buildLongLivedTokenUrl`).
- **Meta App credentials and Instagram product credentials are SIBLINGS, not equals.** Any Meta App that hosts the Instagram product carries two distinct credential pairs that are easy to confuse:
  - `META_APP_ID` / `META_APP_SECRET` (App Settings → Basic) — sign every webhook (`X-Hub-Signature-256` HMAC) and authorize app-level Graph API calls (e.g. `POST /{appId}/subscriptions`).
  - Instagram `client_id` (parsed from `INSTAGRAM_AUTHORIZE_URL`) / `INSTAGRAM_APP_SECRET` (Meta App Dashboard → Instagram → API setup with Instagram Business Login) — used ONLY for the Instagram OAuth code→short and short→long token exchanges.
  Confusing the two yields opaque "Invalid platform app" / "redirect_uri mismatch" errors from `api.instagram.com/oauth/access_token`. The OAuth script's startup validation enforces both env vars and never falls back to `META_APP_ID` / `META_APP_SECRET`. See [Meta setup guide](./docs/META-SETUP-GUIDE.md) and [Setup verification](./docs/features/setup-verification.md).
- **Token type per channel — System User is primarily a WhatsApp permanence concern.** Empirically validated during Stage 3 manual testing:
  - **WhatsApp Cloud API**: System User token is the required path for production-permanent tokens. The Dashboard has no equivalent "Generate Permanent Token" button for WhatsApp; user-derived tokens expire (1h short-lived, ~60d long-lived) and would break prod.
  - **Messenger**: use a **Dashboard-generated Page Access Token** (App Dashboard → Messenger → API setup → Generate Token for the Page). It is long-lived and effectively permanent so long as the issuing user retains Page admin. System User tokens are NOT accepted on the subscribed_apps endpoint — Meta's [`/{page-id}/subscribed_apps` reference](https://developers.facebook.com/docs/graph-api/reference/page/subscribed_apps/) explicitly requires "*A Page access token requested by a person who can perform CREATE_CONTENT, MANAGE, or MODERATE task on the Page*". The load-bearing phrase is "requested by a person": System User tokens are app-installed (not user-initiated) and don't satisfy that gate regardless of what scopes they carry or what Page-asset roles the System User has. The Dashboard "Generate Token" button mints a Page token from the logged-in user, which does. Symptom of the wrong token type: `subscribe_messenger_page_app failed: HTTP 403, code 210` ("Subject not visible") in `register-webhooks.ts`. `MESSENGER_PAGE_ACCESS_TOKEN` should be `type: PAGE` with `profile_id` equal to the Page id per Graph API `debug_token` — verify with `GET /debug_token?input_token=...&access_token={appId}|{appSecret}` if subscription POSTs fail.
  - **Instagram (Business Login)**: no permanent path exists. The OAuth long-lived token is ~60 days max and must be refreshed before expiry (Stage 10 will likely automate this; for now, manual rerun of `npm run setup:oauth:instagram`).
- **`MESSENGER_LOGIN_CONFIG_ID` + `scripts/setup/oauth-messenger.ts` is the canonical path to a scope-controlled Messenger Page Access Token.** App Dashboard → Messenger → "Generate Token" mints a Page token from the logged-in user's current scope grant, which is fine if those scopes already cover what you need — but it cannot add scopes the user hasn't granted (you'll get a token missing e.g. `pages_read_engagement` / `pages_manage_metadata` that 200s the audit step in `verify-messenger` but later 403s on `subscribe_messenger_page_app` with "Subject not visible"). `setup:oauth:messenger` runs the Facebook Login for Business flow against a `config_id` you create in App Dashboard → Facebook Login for Business → Configurations, which bundles the EXACT scope set server-side. Authorize URL: `https://www.facebook.com/v{N}/dialog/oauth?client_id=&config_id=&redirect_uri=&response_type=code&state=` (note: `config_id` REPLACES `scope=`; don't pass both). Code → User Token: `GET graph.facebook.com/v{N}/oauth/access_token?client_id=&client_secret=&redirect_uri=&code=`. User Token → Page Token: `GET /me/accounts` returns one entry per managed Page, each with its own `access_token` (that's the Page token). Page tokens derived from long-lived User tokens are effectively non-expiring as long as the issuing user retains Page admin.
- **`INSTAGRAM_AUTHORIZE_URL` is the single source of truth for the Instagram OAuth redirect URI and client_id.** The user pastes the embed authorize URL from Meta App Dashboard → Instagram → API setup with Instagram Business Login → "Authorize this app for Instagram business"; `oauth-instagram.ts` parses `client_id`, `redirect_uri`, and (when present) `state` out of it via `parseAuthorizeUrl`, and never constructs an authorize URL itself in the live flow. Meta's Dashboard embed URL omits `state` — when absent, `generateState()` produces a fresh 16-byte hex nonce and `withState()` appends it to the URL before opening the browser. Earlier CLI overrides for the redirect URI were removed when the env var became the only input; only `--reveal` and `--help` remain. The unused `buildAuthorizeUrl` / `IG_SCOPES` / `validateRedirectUri` helpers were removed in the post-Stage-3 dead-code sweep — the script no longer constructs authorize URLs or scope lists internally.
- **`registerShutdown` from `scripts/lib/console.ts` is the official cleanup-hook surface.** Do not install your own `process.on('SIGINT')` in new setup/capture scripts. The shared registry runs all hooks in parallel with a 5-second budget and avoids races between readline cleanup and per-script handles (tunnel, server). Earlier code had competing handlers; the registry pattern is the resolution.
- **`NGROK_DOMAIN` is required (no longer optional).** Three Meta Dashboard webhook callback URL registrations (WhatsApp, Messenger, Instagram) plus the Instagram Business Login OAuth redirect URI are all anchored to a single public domain. Without a pinned static domain (free tier includes one at https://dashboard.ngrok.com/cloud-edge/domains), every run forces re-registering all four entries in the Dashboard — there is no Graph API to atomically update a registered callback URL. `loadConfig` validates that the value is a bare hostname (no scheme, no path, contains a `.`) and throws otherwise. `TunnelOptions.domain` is also required across `scripts/lib/tunnel.ts` callers; all setup/capture scripts read it from `loadConfig().ngrokDomain` (or accept a `--ngrok-domain=<x>` CLI override).
- **ngrok free-tier static domains end in `.ngrok-free.dev`; paid-tier in `.ngrok.app` / `.ngrok-free.app`.** `NGROK_DOMAIN` is the bare hostname (no `https://`, no path) — the ngrok SDK adds the scheme. Don't copy `.ngrok-free.app` placeholders from older docs verbatim; the ngrok Dashboard at https://dashboard.ngrok.com/cloud-edge/domains is the source of truth for your actual reserved domain.

## Conventions

- TypeScript ESM (`"type": "module"`). Internal imports of `.ts` files use `.js` specifiers (Node ESM resolution: `import { createApp } from './http/app.js'`).
- Node `>=20`. Vitest for tests; `node-mocks-http` for Express handler unit tests; `supertest` for integration tests against the real Express app.
- Structured logging with `pino`. Include enough fields to debug webhook delivery, signature failures, channel routing, and (later) chat-endpoint failures. The integration tests assert specific `traceMarker` values — `inbound.whatsapp`, `inbound.messenger`, `inbound.instagram`, `inbound.unknown` — preserve those when extending the dispatcher.
- Capture artifacts (`.captures/`) and `.env` are gitignored and may contain phone numbers, tunnel URLs, message content, and secrets — manually redact before promoting captures into `tests/fixtures/meta/captured/` (Stage 3 workflow — see [Payload capture](./docs/features/payload-capture.md)).
- Stage 3 setup and capture scripts are run via `node --import tsx scripts/...` (see `package.json` scripts). Each script calls `import 'dotenv/config'` at the top to load `.env`, then `loadConfig()` from `src/config/loader.ts`. The `oauth-instagram.ts` script reads `INSTAGRAM_AUTHORIZE_URL`, `INSTAGRAM_APP_SECRET`, and `NGROK_DOMAIN` directly (bypassing `loadConfig`) because it doesn't need `META_VERIFY_TOKEN` or `CHAT_ENDPOINT_URL` to run. It deliberately does NOT consume `META_APP_ID` / `META_APP_SECRET` — Instagram product OAuth uses its own credential pair (see the load-bearing constraint on Meta vs. Instagram credentials). `oauth-messenger.ts` likewise reads env directly, but DOES consume `META_APP_ID` / `META_APP_SECRET` (the Messenger OAuth flow authenticates as the Meta App itself) plus `MESSENGER_LOGIN_CONFIG_ID` and `NGROK_DOMAIN`.
- New feature docs follow `docs/features/<name>.md`: what it does, how it works, code files, configuration, known limitations.
