# Architecture

## Overview

`meta-ai-agent` receives Meta webhooks for WhatsApp Cloud API, Facebook Messenger, and Instagram DMs on a single endpoint, verifies signatures against the raw request body, routes by channel, parses each channel's payload into a normalized message, buffers inbound bursts, calls a developer-provided chat endpoint, and delivers replies through the appropriate channel adapter using channel-aware ordered delivery with per-channel pacing and transient retry.

The package stays transport/orchestration focused: the developer owns the chat endpoint and any model, prompt, account, or product logic behind it.

Conversation state, the buffer scheduler, and the rate-limiter's counter store are **dual-path**: in-memory by default (single-process, lost on restart) or Redis-backed (durable, multi-replica) when `REDIS_URL` is set — the Stage 10 production persistence path, with a BullMQ buffer scheduler. See [Persistence](./features/persistence.md) and [Rate limiting](./features/rate-limiting.md).

## Single Meta App, three products, one webhook URL

A single Meta App can host all three messaging products at once. All three deliver to the same callback URL and use the same `X-Hub-Signature-256` HMAC scheme — but they do **not** all sign with the same secret: WhatsApp and Messenger sign with `META_APP_SECRET`, while Instagram signs with the Instagram product's own `INSTAGRAM_APP_SECRET` (verified against the live API 2026-05-20). The verifier tries all configured secrets — see [Webhook security](./features/webhook-security.md). They are distinguished by the top-level `object` field on the webhook body:

| `object` value | Product | User ID | Business ID | Token type |
| --- | --- | --- | --- | --- |
| `whatsapp_business_account` | WhatsApp Cloud API | `wa_id` (E.164) | `phone_number_id` | System User Access Token |
| `page` | Messenger Platform | PSID | Page ID | Page Access Token |
| `instagram` | Instagram (Business Login) | IGSID | IG User ID | Instagram User Access Token |

`objectToChannel(req.body.object)` in `src/http/app.ts` maps each value to a `Channel` (`'whatsapp' | 'messenger' | 'instagram' | 'unknown'`). The `dispatchWebhook` function uses that to log/route. Anything else returns `unknown` and is warn-logged but still ACKed with 200.

## Data flow

```
Meta POST /webhook (any channel)
   |
   v
express.json({ verify }) -- captures req.rawBody before parse
   |
   v
traceMiddleware -- validate/echo x-trace-id (injection-guarded); pino child on res.locals
   |
   v
createMetaSignatureVerifier -- HMAC-SHA256 over rawBody, constant-time compare
   |           \-- onReject(reason) -> webhook_secret_rejections_total{reason} (missing_signature|mismatch|no_raw_body)
   | 401 invalid_signature / 400 raw_body_unavailable
   v
ACK 200 EVENT_RECEIVED  (sent first; processing is async, fire-and-forget)
   |
   v
dispatchWebhook(req.body, {metrics, traceId, requestLogger})
   |       \-- objectToChannel(req.body.object) -> whatsapp | messenger | instagram | unknown
   |       \-- webhook_received_total{channel,result} (+ webhook_parse_failures_total on the defensive catch)
   v
parseMetaWebhook(req.body) -> ParseResult { messages, statuses }
   |  (non-throwing; dispatcher catches any unexpected throw and logs at error)
   v
per-message log (inbound.message; warn iff type==='unknown')
per-status  log (inbound.status)
summary     log (inbound.{channel}; warn for unknown)
   |
   v
ConversationAgent -- per parsed message: handleInbound; per status: handleStatus
   |  (sequential awaited loop, preserving intra-webhook order; agent OPTIONAL on createApp)
   |  (traceId + child logger threaded in; metric counters on every path)
   |
   |  handleInbound: echo filter -> SETNX dedupe (claimInboundHandle)
   |               -> IdentityResolver.resolve (fail-open, once per conversation; identity_lookup_total)
   |               -> append to buffer (state: buffering) -> (re)arm flush timer
   v
buffer flush (timer; state: processing)
   |  snapshot+clear buffer -> ChatClient.complete(ChatRequest {contact?}) (chat_dispatch_duration_seconds)
   v
normalizeChatResponse -> ChatAction[]  -> buildOutboundItems (capability-gated)
   |
   v
ordered outbound queue (state: sending) -> ChannelAdapter (see Outbound path below)
   |  pre-send pacing (LimitTracker.acquireSendSlot, fail-open) on message/reply/template/media/reaction
   |  outbound_send_total / _duration_seconds per send; recordOutbound bumps track-only hour/day counters; acquire_send_slot_delay_seconds
   |  on a SYNC send error: classifyError -> transient (retry w/ backoff) | window_closed (WhatsApp template re-prompt) | permanent (skip+advance)
   |  WhatsApp advances on sent/delivered status (handleStatus); Messenger/IG advance on send
   v
handleStatus / handleStatusImpl / handleReadWatermark -> StatusTracker (status_callback_total; observability only)
   |  WhatsApp per-message (failed -> classifyStatusErrorCode async retry/reprompt/skip; errorCategory bucket); Messenger/IG read watermark -> translated to sent message ids
   |  late status logs carry conversationTraceId (chained to the originating inbound webhook)
   v
queue complete / silence / chat error -> state: idle

Operational surface (separate request paths):
   GET /health, GET /ready                 (always on, unauth)
   GET /metrics                            (token-gated; renderPrometheus(collector.snapshot()))
   GET /admin/conversations/:key           (token-gated; PII-redacted, ?reveal=true)
   GET /admin/status/:messageId            (token-gated; PII-redacted, ?reveal=true)
   GET /admin/queue                        (token-gated; { kind, stats? } from the scheduler, timeout-raced)
   GET /admin/dedupe?messageId=<id>        (token-gated; { messageId, present, ttlSeconds? }; 400 if messageId missing)
   -- /metrics + /admin/* are GUARDED AT REGISTRATION (unmounted -> 404 when ADMIN_API_TOKEN unset)
```

Per-conversation flows are serialized by a per-key async lock (`runExclusive`) so
concurrent inbounds/statuses for one conversation can't clobber the
clone-on-write store; different keys still run concurrently. See
[Conversation state](./features/conversation-state.md).

### Outbound path (wired to the conversation agent)

```
ConversationAgent.sendNext   -- ordered, channel-aware advancement; typing injected before text
   |  sendText / sendTypingIndicator / markRead / sendReaction / sendMedia (+ WhatsApp sendTemplate)
   v
ChannelAdapter            -- one of WhatsAppClient / MessengerClient / InstagramClient
   |  (owns channel body shapes: messaging_product / recipient / sender_action / template / media attachment)
   v
GraphClient.request       -- shared transport: versioned URL, Bearer header,
   |                          retry/backoff (429 + network always; 5xx only if idempotent —
   |                          POST sends are non-idempotent, so a 5xx is NOT retried)
   v
Meta Graph API            -- graph.facebook.com (WhatsApp/Messenger) or graph.instagram.com (Instagram)
```

The conversation agent drives the per-channel clients at runtime, including the
uniform `sendMedia` (the agent infers the media kind from the action's MIME and
dispatches without channel branching). See
[Ordered delivery](./features/ordered-delivery.md),
[Outbound clients](./features/outbound-clients.md),
[Media send](./features/media.md), and
[WhatsApp templates](./features/templates.md).

Meta retries any non-2xx response with exponential backoff for up to 7 days, then permanently drops the event. There is no replay API. The handler is the dead-letter queue. The POST route sends the 200 before invoking the dispatcher.

## Key design decisions

- **Single endpoint for three products.** Routing by `object` is simpler and matches Meta's design (one App, one webhook URL, one secret). It also keeps signature verification uniform across channels.
- **Raw-body capture before parsing.** `express.json({ verify })` is the only safe place to copy the bytes Meta signed — by the time the JSON body lands on `req.body`, whitespace and key order are gone. Any future middleware that needs to read the body must respect `req.rawBody`.
- **ACK before dispatch.** The POST handler sends `200 EVENT_RECEIVED` before calling `dispatchWebhook`. Meta's retry behavior (7 days of exponential backoff) means slow processing would queue thousands of duplicate deliveries. This is non-negotiable.
- **Identity is a tuple, not a unified user.** Meta does not link `wa_id`, PSID, and IGSID. The package models identity as `(channel, channelScopedId)` and leaves cross-channel merging to the developer's identity resolver (Stage 6 — via the `unifiedContactId` the resolver returns; this package never synthesizes one). See [Identity resolution](./features/identity-resolution.md).
- **Adapters expose capabilities, not a uniform feature set.** All three clients share the `ChannelAdapter` *signature*, but `supports('template')` returns `false` for Messenger and Instagram (it is a WhatsApp concept). The conversation agent branches on capability (via `buildOutboundItems` gating) rather than assuming features are uniformly available. See [Outbound clients](./features/outbound-clients.md) for the full matrix.
- **Per-conversation serialization, not a global lock.** The Stage 5 store is pass-by-value with last-write-wins, so two concurrent flows for one conversation can clobber each other's record (a confirmed message-dropping race). The agent serializes per key via a promise tail (`runExclusive`); different keys still run concurrently. Entry points acquire the lock; internal helpers stay lock-free. See [Conversation state](./features/conversation-state.md).
- **Segmented per-conversation locking + interrupt/rebatch.** The flush no longer holds the per-key lock across the (slow) chat-endpoint call — it acquires the lock only around its record read-modify-write segments and releases it for the network call, which it makes against a private snapshot that mutates no record. WHY: a fast typist's second message could not interrupt a flush that held the lock through the chat call, so two close-together messages produced two separate replies. Now a message arriving mid-flush aborts the in-flight chat call (a per-key `AbortController`) and rebatches into ONE combined chat turn, while every record mutation stays a complete locked read-modify-write so the lock's race-safety is preserved. A `MAX_REPROCESS` cap guarantees forward progress (a never-ending stream eventually gets a committed, un-abortable flush). See [Conversation state](./features/conversation-state.md), [Message buffering](./features/message-buffering.md), and [Ordered delivery](./features/ordered-delivery.md).
- **Channel-aware queue advancement.** WhatsApp emits per-message delivery statuses, so its outbound queue waits for a `sent`/`delivered` webhook (`on_status`) with a delivery-timeout fallback. Messenger/Instagram have no reliable per-message delivery callback, so their queue advances on the successful send response (`on_send`). One queue abstraction, two confirmation rules. See [Ordered delivery](./features/ordered-delivery.md).
- **Outbound 5xx is not retried on POST (double-send safety).** The shared `GraphClient` retries `429` and pre-response network failures on any method, but a `5xx` only when the request is idempotent. Sends are POST and non-idempotent, so an ambiguous post-acceptance 5xx is surfaced rather than retried. See [Outbound clients](./features/outbound-clients.md).
- **Admin surface guarded at registration (Stage 6).** `/metrics` and `/admin/*` are mounted only when `ADMIN_API_TOKEN` is set; unset → the routes don't exist (404), rather than mounted-and-401. Returning 404 means an un-configured admin surface is indistinguishable from a non-existent route, so a token-less deploy never advertises that an admin surface is there. See [Operational visibility](./features/operational-visibility.md).
- **Admin output is allow-list redacted, fail-closed (Stage 6).** The redactors copy only known-safe structural fields and mask/drop content by default, so a field added to a record/message later is omitted from the masked view until someone deliberately allow-lists it — new PII fails closed. `?reveal=true` (authenticated) is the escape hatch. Chosen over a deny-list precisely because a deny-list fails open. See [Operational visibility](./features/operational-visibility.md).
- **Identity enrichment is fail-open (Stage 6).** A configured `USER_LOOKUP_URL` is best-effort: any failure resolves to "no contact, proceed" and never blocks delivery. Status watermark reads are observability-only and never touch the queue. See [Identity resolution](./features/identity-resolution.md) and [Status tracking](./features/status-tracking.md).
- **Inbound media hydration is opt-in.** A WhatsApp inbound image arrives as a bare media id that needs an authenticated 2-hop Graph fetch (with the WhatsApp access token, which the chat endpoint does not hold) — so the transport, which does hold the token, can download it once on the flush path and hand the endpoint a ready base64 `data:` URL on `message.media.dataUrl`. It is OFF by default (`INBOUND_MEDIA_DOWNLOAD`) because base64 inflates the request body ~33% over the raw bytes; a per-attachment `INBOUND_MEDIA_MAX_BYTES` cap (default 5 MiB) bounds the cost and the whole path is fail-open. See [Media hydration](./features/media-hydration.md).
- **Dual-path persistence, one shared client (Stage 10).** Conversation state, the buffer scheduler, and the rate-limit counter store swap to Redis-backed implementations when `REDIS_URL` is set — durable across restarts and coordinated across replicas (real `SET NX` dedupe, BullMQ delayed flushes, a Lua-atomic pacing slot). One injected ioredis client is shared across the conversation store + counter store (the BullMQ scheduler owns its own connections); the shared client is BORROWED, so those stores' `close()` is a no-op and the runtime owns the single disconnect. See [Persistence](./features/persistence.md).
- **Transient retry is double-send-safe on the SYNC path (Stage 10).** The `LimitTracker`'s `classifyError` only re-sends a failed POST when Meta is known NOT to have processed it — a pre-response network failure, a 429, or a Meta rate-limit error code — and treats any 5xx (or opaque error) as permanent, mirroring the GraphClient's no-retry-on-5xx-POST rule. The messages endpoint has no idempotency key, so a re-send of a possibly-delivered request would double-send. WhatsApp out-of-window failures instead re-prompt the chat endpoint once for a template. See [Rate limiting](./features/rate-limiting.md).
- **The double-send rule is INVERTED on the ASYNC `failed`-status path (limits wave).** A WhatsApp `failed` *delivery-status* webhook is Meta's definitive statement the message did NOT reach the user, so `handleStatusImpl` can safely retry a transient-classified async failure (no double-send) — the opposite of the sync 5xx rule, and intentional. The async loop is bounded by the dedicated `asyncFailRetryCount` (separate from `retryCount`, which the success tail clears) so a recurring async failure can't loop forever, and the dead handle is reset before a clean re-send. See [Ordered delivery](./features/ordered-delivery.md#async-failure-from-a-failed-delivery-status).

## Known limitations (Stages 1–10)

- **Persistence is dual-path; the Stage 6 stores are still in-memory.** Conversation state, dedupe, the buffer scheduler, and the rate-limit counter store are Redis-backed when `REDIS_URL` is set (`RedisConversationStore` + `BullMqBufferScheduler` + `RedisLimitCounterStore`, sharing one client), else in-memory. But the Stage 6 metrics collector / status tracker / contact cache (`InMemoryMetricsCollector`, `InMemoryStatusTracker`, `InMemoryContactStore`) stay **in-memory in both paths** — lost on restart, unbounded apart from the metrics cardinality cap (`__overflow__`); their Redis-backed swaps with TTL eviction are deferred. The in-memory store/scheduler/counter remain available for tests, local runs, and single-replica deploys.
- Identity resolution is wired (Stage 6) but **fail-open and optional** — `ChatRequest.contact` is populated only when `USER_LOOKUP_URL` is set and the lookup succeeds, else undefined. Metrics are wired (Stage 6); the `/ready` Redis check is now a real timeout-bounded ping (Stage 10) when a client is injected, but per-dispatch webhook logs still emit channel-scoped ids at `info`, the webhook-signature-rejection metric is not wired, and the identity metric is coarse — see [Known gaps](./KNOWN-GAPS.md).
- A failed outbound send is now classified (Stage 10): a known-safe transient failure (network / 429 / Meta rate-limit code) is retried with backoff up to a cap, a WhatsApp `window_closed` failure re-prompts the chat endpoint once for a template, and everything else (incl. any 5xx on the SYNC path — double-send safety) is skipped + advances. A WhatsApp `failed` *delivery-status* webhook is routed the same three ways (limits wave) via `classifyStatusErrorCode`, but with the double-send rule inverted (a `failed` status is definitive non-delivery, so a transient retry is safe — bounded by `asyncFailRetryCount`). A media send Meta rejects (e.g. a non-PDF / oversized Instagram `file`) is permanent → skip. The 24h window is ENFORCED on WhatsApp via the out-of-window re-prompt (it was previously tracked-but-not-enforced); Messenger/IG have no reliable out-of-window mechanism.
- Outbound clients cover text, typing, read receipts, reactions, (WhatsApp) templates, and (Stage 7) media send across all three channels (`supports('media_send')` is true everywhere). Media is fully buffered in memory (no streaming) and the clients do not validate MIME/size before sending — Meta rejects an unsupported asset, caught fail-soft. See [Media send](./features/media.md).
- Pacing **gating** is **per-second only** — the per-hour/per-day counters are **track-only** (advisory warn/error logging via `recordOutbound`, fail-open), not a gating cap, and there is no WhatsApp conversation-unit accounting or Prometheus metric for a crossing (Meta's daily tier limits and the IG hourly throughput model are tracked by Meta, not enforced here). The Instagram client's own coarse ~100ms in-process floor still runs **in addition** to the `LimitTracker` (double-paced; both conservative). See [Rate limiting](./features/rate-limiting.md) and [Known gaps](./KNOWN-GAPS.md).
- Fixtures under `tests/fixtures/meta/` are documentation-derived. Real Meta payloads always differ in subtle ways; the Stage 3 `npm run capture:guided` tooling exists for this exact purpose — promoted captures land under `tests/fixtures/meta/captured/` after manual redaction.
- The Stage 9 examples and REPL are dev tooling, not production code. The in-repo examples import the chat-contract types via relative `src/` paths (a real consumer imports them from the published package); the showcase-bot keeps its own field-for-field copies in a separate package (on the Vercel AI SDK — `ai`, `@ai-sdk/anthropic`, `@ai-sdk/openai` — plus media/STT deps, outside the root tsc/test suite). The REPL fakes the channel adapters (no real Meta sends) and uses low buffer timeouts for snappiness; the REPL, `example-dev`, and the showcase bot have no automated tests in the root suite (only the pure handlers and webhook builders are unit-tested; the showcase-bot is verified manually/live + its own `npm run typecheck`) — see [Known gaps](./KNOWN-GAPS.md) and [`examples/README.md`](../examples/README.md).
- Token-format checks are **advisory warnings, not hard rejects** (Stage 10) — Meta token formats vary, so `loadConfig` warns rather than throws on an unexpected shape (`REDIS_URL`, by contrast, IS hard-validated). See [Persistence](./features/persistence.md), [Rate limiting](./features/rate-limiting.md), and [Known gaps](./KNOWN-GAPS.md).
- See [Known gaps](./KNOWN-GAPS.md) for items intentionally deferred (CTWA pricing/conversation blocks, order/contact attachments, IG `story_mention.id` semantics, Dashboard programmatic config gaps, token refresh automation, the in-memory metrics/status/contact stores).
